    - Test 2: When upserting a URI that does not exist in db, the function indicates it was new
    - Test 3: When upserting a URI that already exists in db, the function indicates it was existing
  </behavior>
  <action>
1. In `models.rs`, add two fields to `FetchSummary`:
   ```rust
   pub struct FetchSummary {
       pub count: u64,
       pub done: bool,
       /// Number of posts that were newly inserted.
       pub new_count: u64,
       /// Number of posts that already existed and were updated.
       pub existing_count: u64,
   }
   ```

2. In `db.rs`, change `upsert_post` return type from `Result<(), ExtractorError>` to `Result<bool, ExtractorError>` where `true` = new insert, `false` = existing/replaced. Implementation: call `db_has_uri(conn, uri)?` before the INSERT OR REPLACE. If it returns true, the post exists (return Ok(false) after upsert). If false, it is new (return Ok(true) after upsert). This is cleaner than relying on `conn.changes()` semantics.

3. In `client.rs` `fetch_all_posts`, update the loop body where `upsert_post` is called (around line 280):
   - Change `crate::db::upsert_post(conn, &record.uri, did, &text, &created_at, &raw_json)?;` to capture the return: `let is_new = crate::db::upsert_post(conn, &record.uri, did, &text, &created_at, &raw_json)?;`
   - Add mutable counters `let mut new_count = 0u64;` and `let mut existing_count = 0u64;` alongside the existing `let mut count = 0u64;`
   - After each upsert: `if is_new { new_count += 1; } else { existing_count += 1; }`
   - Update ALL return sites of FetchSummary (there are 3: two early returns for incremental-stop and since-cutoff, plus the final return) to include the new fields: `FetchSummary { count, done: true, new_count, existing_count }`

4. Also in `client.rs`, update the incremental-stop early return (around line 259-261). Currently when `db_has_uri` returns true, it returns immediately. This is correct -- but note that the "hit a known URI" record itself is neither new nor existing in the count (it was skipped). The counts only reflect records actually processed through upsert_post. No change needed to this logic beyond adding the new fields to the return.

5. Update the existing tests in `client.rs` `mod tests` that construct `FetchSummary` or check its fields. The `test_fetch_all_posts_db_roundtrip` test does not construct FetchSummary directly so should be unaffected. Verify all tests still compile.

  <action>
1. In `client.rs`, add an optional progress callback parameter to `fetch_all_posts`. The function currently has `#[allow(clippy::future_not_send)]` because `rusqlite::Connection` is not Send. A closure callback must also be non-Send compatible. Use a concrete type alias to keep the signature clean:

   Change the signature from:
   ```rust
   pub async fn fetch_all_posts(
       &mut self,
       did: &str,
       since: Option<chrono::DateTime<chrono::Utc>>,
       conn: &rusqlite::Connection,
   ) -> Result<FetchSummary, ExtractorError>
   ```
   to:
   ```rust
   pub async fn fetch_all_posts(
       &mut self,
       did: &str,
       since: Option<chrono::DateTime<chrono::Utc>>,
       conn: &rusqlite::Connection,
       on_progress: Option<&dyn Fn(u64)>,
   ) -> Result<FetchSummary, ExtractorError>
   ```

   The `on_progress` callback receives the running total count after each record is processed. Inside the loop, after incrementing `count`, call:
   ```rust
   if let Some(cb) = on_progress {
       cb(count);
   }
   ```

2. In `lib.rs`, update `run_extraction` to accept and pass through the callback:

   Change the signature from:
   ```rust
   pub async fn run_extraction(
       handle: &str,
       db_path: &std::path::Path,
       since: Option<chrono::DateTime<chrono::Utc>>,
   ) -> Result<models::FetchSummary, error::ExtractorError>
   ```
   to:
   ```rust
   pub async fn run_extraction(
       handle: &str,
       db_path: &std::path::Path,
       since: Option<chrono::DateTime<chrono::Utc>>,
       on_progress: Option<&dyn Fn(u64)>,
   ) -> Result<models::FetchSummary, error::ExtractorError>
   ```

   Pass `on_progress` through to `bsky_client.fetch_all_posts(...)`:
   ```rust
   bsky_client.fetch_all_posts(&did, since, &conn, on_progress).await
   ```

3. Ensure both functions retain `#[allow(clippy::future_not_send)]` -- the `&dyn Fn(u64)` reference is not Send, which is expected since this runs on a single-threaded tokio runtime.

4. Add doc comments to the new parameter on both functions:
   - `/// The `on_progress` callback, if provided, is invoked with the running`
   - `/// total of processed records after each post is stored.`
